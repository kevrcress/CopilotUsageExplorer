## Findings

### IV-001 [SECURITY] [CRITICAL]
Description: Electron main-process IPC handlers pass the renderer-supplied `installId` string directly into `path.join()` to build a filesystem root, with no validation against the fixed `PRODUCT_DIRS` allowlist (`Code`, `Code - Insiders`, `VSCodium`, `Cursor`) that the design intends. This is a path-traversal / arbitrary-directory-read primitive reachable from the renderer via `ipcMain.handle`.
Evidence: `apps/electron/src/main/discovery.ts:60-62` (`workspaceStorageDir(productId)` does `path.join(userDataRoot(), productId, 'User', 'workspaceStorage')` with no allowlist check); `apps/electron/src/main/discovery.ts:160-181` (`listBucketRefs`/`readBucket` call `workspaceStorageDir(installId)` where `installId` is only checked with `typeof installId !== 'string'` at `apps/electron/src/main/index.ts:54-62`); `apps/electron/src/main/discovery.ts:275-279` (`startWatch` same pattern).
Impact: A compromised or buggy renderer (e.g., via a future feature, a dependency supply-chain issue, or a dev-tools-open user pasting into the console) can pass `installId = "../../../../etc"` or an absolute-like traversal string and have the main process `readdir`/`readFile` arbitrary paths on the user's machine, then ship file contents back over IPC to the renderer. `contextIsolation`/`nodeIntegration` are correctly configured (`apps/electron/src/main/index.ts:112-113`) but do not mitigate a main-process path-traversal bug — the renderer already has a sanctioned channel to request arbitrary main-process file reads.
Recommendation: Validate `installId` against `PRODUCT_DIRS.map(p => p.id)` (or an equivalent enum) at the top of `listBucketRefs`, `readBucket`, and `startWatch` (or once centrally in the `ipcMain.handle` wrappers in `index.ts`), rejecting/returning empty on any non-allowlisted value. Apply the same allowlist check to `ref.hash`/`ref.session` path segments if they can contain `..` (currently only `typeof` is checked at `index.ts:59-61`).

### IV-002 [DRY] [MAJOR]
Description: `dexie-cache.ts` and `localstorage-prefs.ts` are byte-for-byte identical across `apps/web/src/adapters/` and `apps/electron/src/renderer/src/adapters/` (confirmed via `diff`, zero output). This was a deliberate, disclosed decision (DD-303) to avoid cross-touching packages during parallel Phase 2/3 work, but it is now a standing maintenance liability: a bug fix or schema change in one copy silently does not propagate to the other.
Evidence: `apps/web/src/adapters/dexie-cache.ts` vs `apps/electron/src/renderer/src/adapters/dexie-cache.ts` (identical); `apps/web/src/adapters/localstorage-prefs.ts` vs `apps/electron/src/renderer/src/adapters/localstorage-prefs.ts` (identical). Also `apps/vscode-ext/webview/dexie-cache.ts` is a third, separately-maintained copy per DD-201.
Impact: Three independent copies of the same Dexie schema/version logic exist. A future Dexie schema migration (e.g. adding an index) applied to one copy but not the others will cause silent cache-format drift between hosts, surfacing as host-specific bugs that are hard to correlate.
Recommendation: Given the plan's parallel-phase constraint has now resolved (all phases complete, nothing committed), hoist `dexie-cache.ts` and `localstorage-prefs.ts` into a shared location (e.g. `packages/ui/src/adapters/` or a new `packages/host-common`) before this branch is committed/merged, since core purity rules out putting Dexie there. This is the natural point to consolidate — deferring it risks the duplication becoming permanent once each app's tests/build depend on its own copy.

### IV-003 [ERROR-HANDLING] [MAJOR]
Description: `readSessionBucket`/`listBucketRefs` (Electron) and `discoverWorkspaces`/`readWorkspaceBuckets` (VS Code) both walk `sessionName`/`hash` values taken from `fs.readdir()` results and join them into paths without checking for `..` or path separators embedded in a crafted directory name. On a normal filesystem this can't happen from real VS Code data, but the same `readSessionBucket`/`readBucket` helpers are reachable with the unvalidated `installId` from IV-001, compounding the risk: an attacker-influenced `installId` reaching a real-seeming `wsDir` could combine with a directory an attacker planted (unlikely without local code exec already, but worth closing off given the layered IPC surface is otherwise well-guarded).
Evidence: `apps/electron/src/main/discovery.ts:132-136` (`readSessionBucket(wsDir, hash, sessionName)` — `hash`/`sessionName` unchecked before `path.join`); `apps/vscode-ext/src/discovery.ts:118-124` (`readSessionBucket(ws, sessionId)` similarly unchecked, though `sessionId` here always comes from a same-process `fs.watch` event, not cross-boundary IPC, which is a materially lower-risk path than the Electron case).
Impact: Low standalone risk (requires local code execution to plant a malicious directory name), but it's the second half of the same missing-validation pattern flagged in IV-001, and defense-in-depth at the IPC boundary is the standard mitigation once one path-traversal gap is found.
Recommendation: Add a shared `assertSafeSegment(name: string)` helper (reject entries containing `..`, `/`, `\`) applied at the point where `readdir` results are treated as trusted path segments, in both `apps/electron/src/main/discovery.ts` and `apps/vscode-ext/src/discovery.ts`. Low effort, closes the residual gap alongside the IV-001 fix.

### IV-005 [DOCUMENTATION] [MINOR]

### IV-006 [TEST-COVERAGE] [MINOR]
Description: No new automated tests were added for any of the ~1,750 lines of new host-specific code in `apps/vscode-ext` (protocol/discovery/watcher/file-cache/update-check/adapters) or `apps/electron` (discovery/main/preload/adapters/ingest-controller). This is explicitly acknowledged in the plan's Testing Approach ("core tests are the regression net for both hosts") and the changes log, so it is a disclosed, deliberate scope choice rather than an oversight — flagged here per dimension 8 for visibility, not as a plan deviation.
Evidence: `find apps -iname "*.test.*"` returns nothing; changes log Phase 2/3/4 "Tests:" lines confirm only the pre-existing 14 core tests run. Highest-value untested logic: `compareVersions` in `apps/vscode-ext/src/update-check.ts:12-20` (pure function, trivially unit-testable, currently only exercised manually); `scripts/sync-version.mjs` (exercised manually per the changes log but has no `packages/core`-style unit test even though it's dependency-free and pure).
Impact: Host-specific logic (discovery path construction, protocol chunking thresholds, version comparison) has no regression net; future edits to `compareVersions` or `sync-version.mjs` could silently break update-check or CI version-gating without a failing test.
Recommendation: At minimum, add a small Vitest suite for the two pure, dependency-free units that don't require mocking `vscode`/`electron`: `compareVersions()` (extract if needed) and `sync-version.mjs`'s stamp/check/expect logic. This is cheap, high-signal, and doesn't require the manual-verification-only VS Code/Electron runtime harness.

### IV-007 [API-USAGE] [MINOR]
Description: The VS Code webview CSP correctly locks `script-src` to `${webview.cspSource}` (no inline scripts, boot data passed via a `data-*` attribute instead — a good practice, called out explicitly in DD-203), but `style-src` includes `'unsafe-inline'` to accommodate Recharts' inline `style` attributes. This is a reasonable, disclosed tradeoff (documented at `apps/vscode-ext/src/extension.ts:88-90` and details §5), not a defect, but it does widen the CSP surface beyond the minimum, and there's no code comment at the actual CSP string cross-referencing why (only in a comment above it).
Evidence: `apps/vscode-ext/src/extension.ts:95-101` (CSP array); the `'unsafe-inline'` is only for `style-src`, not `script-src` — correctly scoped.
Impact: None beyond the accepted tradeoff; flagged only because `style-src 'unsafe-inline'` does technically permit CSS-based data exfiltration vectors (e.g. attribute-selector scraping) in a compromised-dependency scenario, though this is a standard, low-severity accepted risk for chart libraries.
Recommendation: No action required beyond what's already documented; consider a follow-up note in `apps/vscode-ext/README.md` "Limitations" section if a stricter CSP is ever desired (e.g. migrating off Recharts inline styles).

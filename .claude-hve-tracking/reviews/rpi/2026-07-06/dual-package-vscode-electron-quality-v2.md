## Findings

### IV2-001 [DRY] [MINOR]
Description: R6.2's discovery-utility hoist to `packages/core/src/node-fs-utils.ts` is substantively complete (isDir/listSubdirs/readWorkspaceFriendlyName/serializeFile/groupByParentDir/mapWithConcurrency all shared), but a small helper and a small constant object were left duplicated across the two apps rather than hoisted alongside them.
Evidence: `errMessage()` is defined identically (3-line `err instanceof Error ? err.message : String(err)`) in three places: `apps/electron/src/main/discovery.ts:73-75`, `apps/vscode-ext/src/discovery.ts:100-102`, `apps/vscode-ext/src/watcher.ts:86-88`. `noopLogger` (`{ info() {}, warn() {}, error() {} }`) is defined identically twice within vscode-ext alone: `apps/vscode-ext/src/discovery.ts:30` and `apps/vscode-ext/src/watcher.ts:10`.
Impact: Minor maintenance friction only — if the error-formatting convention or no-op logger shape ever changes, three (resp. two) call sites must be updated in lockstep. Not a correctness risk; these are pure, trivial, unlikely-to-drift helpers.
Recommendation: Fold `errMessage` into `packages/core/src/node-fs-utils.ts` (or a tiny shared `log-utils.ts` alongside it) and export `noopLogger`/a `Logger` default-instance helper from `apps/vscode-ext/src/logger.ts` so `discovery.ts` and `watcher.ts` both import it instead of each declaring their own. Low priority; not blocking.

### IV2-002 [ERROR HANDLING] [MINOR]
Description: R6.4's conversion of `apps/vscode-ext/webview/main.tsx`'s bootstrap from module-scope statements into an async `main()` function invoked as `void main()` introduces a new failure mode with no handler: `createCache()` now does a dynamic `import('@cue/ui/adapters/dexie-cache')` when `cacheBackend === 'indexeddb'`, and `main()` itself has no top-level try/catch.
Evidence: `apps/vscode-ext/webview/main.tsx:91` (`void main();`) with no `.catch()`; `main()` body (`:30-89`) has no surrounding try/catch; `createCache()` (`:16-22`) can reject if the dynamic import fails (e.g. asset 404, CSP block, or a build/deploy mismatch between `dist/webview` chunks).
Impact: If `createCache()` (or any other awaited step in `main()`, e.g. `initAppStore`) throws, the promise rejection is unhandled and the webview silently renders nothing — no error UI, no console signal surfaced to the user, no fallback to the default backend. This is a narrow but real regression in robustness versus the pre-R6.4 static-import bootstrap, which could not fail this way. Not present before R6.4 since the import was static and resolved at build time.
Recommendation: Wrap `void main()` with `.catch((e) => { console.error(e); /* render a minimal fallback error state or fall back to createGlobalStorageCache */ })`, or add a top-level try/catch inside `main()` around `createCache()` specifically, falling back to `createGlobalStorageCache(bridge)` on dynamic-import failure so a broken indexeddb chunk load can't blank the whole panel.

### IV2-003 [TEST COVERAGE] [MINOR]
Description: Phase 6 (R6.2) added `packages/core/src/node-fs-utils.ts` (6 functions: isDir, listSubdirs, readWorkspaceFriendlyName, serializeFile, groupByParentDir, mapWithConcurrency, including the new bounded-concurrency worker-pool logic) and the `buildHashTree` TTL cache in `apps/electron/src/main/discovery.ts`, with zero new unit tests for any of it.
Evidence: `find packages/core/test apps/vscode-ext/test apps/electron -iname "*.test.ts"` returns only `ingest.test.ts`, `parser.test.ts`, `tokens.test.ts`, `version-compare.test.ts` — no `node-fs-utils.test.ts`, no test referencing `mapWithConcurrency` or `buildHashTree` anywhere in the tree.
Impact: `mapWithConcurrency`'s worker-pool cursor logic (order-preservation across concurrent completions) and the 10s TTL cache's expiry/rebuild boundary are exactly the kind of logic that silently breaks under refactoring without a regression test — both are new, non-trivial, and currently only exercised indirectly via manual smoke runs (per Phase 3/6's verification notes). This was flagged as an expected gap in the assignment and is confirmed here.
Recommendation: Add a small `packages/core/test/node-fs-utils.test.ts` covering `mapWithConcurrency` (order preservation with mixed-latency fake promises, limit >= / < items.length, limit=1 sequential-equivalence) and `groupByParentDir` (empty input, single-level, nested relPath). `buildHashTree`'s TTL behavior is harder to unit test cleanly (real fs + real Date.now()) — consider extracting the cache-hit/miss decision into a pure function taking `now` as a parameter, or defer as an acceptable manual-only check given the file lacks any test infra today.

### IV2-004 [DOCUMENTATION] [MINOR]

Impact: Low — the comment is directional/illustrative, not load-bearing, and doesn't cite a symbol or line number that could be "wrong" in a stronger sense. This exact gap was already called out as a Minor in the prior PR review (`Documentation and Operations` section) but was not routed to any Phase 6 work item and remains unfixed today.
Recommendation: Update the comment to `# Vitest: parser / ingest / token-rollup / version-compare / sync-version tests` (or generalize to `# Vitest unit tests` to avoid re-staling on the next test addition). Low priority; does not block.

### IV2-005 [SECURITY] [MINOR]
Description: `.gitignore`'s credential-file hygiene is good but not complete against the full pattern set this validator checks: `*.pfx` and the broader `.env.*` glob are absent (only exact `.env`/`.env.local` are listed). This is a pre-existing gap independently re-confirmed here, not introduced by Phase 5/6, and was already flagged (as Minor) in the prior PR review.
Evidence: `.gitignore:9-13` contains `.env`, `.env.local`, `*.pem`, `*.key`, `*.p12` — no `*.pfx`, no `.env.*` glob form.
Impact: Low — no `.env` usage exists anywhere in the codebase today (confirmed via the Phase-4 Security Hygiene Check and independently re-grepped here with zero secret-pattern hits), and Windows code-signing (`.pfx`) is not currently part of the CI/release pipeline (mac uses `identity: null`, Windows NSIS is unsigned per `electron-builder.yml`). Risk is latent, not active.
Recommendation: Add `.env.*` and `*.pfx` to `.gitignore` proactively, especially before any future Windows code-signing setup that would introduce a real `.pfx` file into local dev environments.
